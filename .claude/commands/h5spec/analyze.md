# H5Spec - Analyze 需求分析

你是 H5Spec Agent，现在执行 **Analyze 阶段**：深入分析业务需求、用户交互和技术可行性。

## 用户输入

```text
$ARGUMENTS
```

You **MUST** consider the user input before proceeding (if not empty).

## 核心任务

基于 `@/h5.md` 中 Analyze 阶段的定义，全面分析用户提供的业务需求、UI/UX 设计和技术可行性。

### 输出目标

生成 `@/docs/ANALYSIS.md` 需求分析文档，包含：

1. **需求分析** - 功能需求列表、非功能需求、业务流程（User Journey）
2. **UI/UX 分析** - 视觉设计评审、交互流程分析、响应式和无障碍要求
3. **技术可行性分析** - 技术栈兼容性、第三方依赖、性能和安全风险评估
4. **用户故事和验收标准** - 明确可执行的验收条件

### 分析维度

**1. 需求分析**
- 功能需求梳理（Feature List）
- 非功能需求（性能、安全、可访问性）
- 业务流程分析（User Journey）

**2. UI/UX 分析**
- 视觉设计稿评审
- 交互流程分析
- 响应式设计要求
- 无障碍访问（a11y）要求

**3. 技术可行性分析**
- 现有技术栈的兼容性
- 第三方依赖的可用性
- 性能和安全风险评估

### 执行要点

1. **询问澄清** - 如需求不明确，使用 AskUserQuestion 工具询问：
   - 目标用户群体和使用场景？
   - 核心功能优先级排序？
   - UI/UX 设计稿或参考案例？
   - 性能指标和兼容性要求？
   - 是否有现有接口文档？

2. **结构化梳理** - 使用 PRD、User Story、Use Case Diagram 等方法
3. **可视化呈现** - 用 mermaid 绘制用户流程图和状态转换图
4. **风险识别** - 提前发现技术难点和潜在风险

### 参考工具

- **Figma/Sketch** - 设计稿检查
- **Zeplin/Avocode** - 设计稿标注
- **Storybook** - 组件预览
- **User Story Mapping** - 需求分析方法

## 输出要求

- 使用中文编写分析文档
- 需求描述具体明确，避免模糊
- 提供用户流程图和交互流程图
- 列出明确的验收标准（Acceptance Criteria）
- 识别并记录技术风险和应对方案
